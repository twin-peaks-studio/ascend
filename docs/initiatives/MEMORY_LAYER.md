## What Is the Memory Layer?

The Memory Layer turns Ascend from a project management tool into a **personal product intelligence platform** — a second brain for your work.

Today, Ascend helps you manage projects and tasks. But all the context around your work — meeting notes, decisions, who said what, why something was built a certain way — lives in your head, scattered across Slack, docs, and email. When you need to recall something from three weeks ago, you're left searching through messages and hoping you remember where to look.

The Memory Layer changes this. It gives Ascend a **memory** — the ability to store, connect, and recall everything you capture about your work. You write things down (captures), tag them with the things they're about (entities), and later ask questions in plain English (chat). The system finds what's relevant and gives you grounded answers, citing exactly where the information came from.

Think of it as having a research assistant who remembers everything you've ever told them, can instantly cross-reference across all your work, and proactively notices patterns you might miss.

## Glossary

These are the core concepts introduced by the Memory Layer. Understanding these terms helps everything else make sense.

| Term | What It Means (Plain Language) | Technical Definition |
|------|-------------------------------|---------------------|
| **Workspace** | A container for all your work in a given context. Like having separate desks for "Work" and "Side Projects." | Top-level isolation boundary. Database table with `type: 'standard' \| 'intelligence'`. All projects, captures, entities, and chat conversations belong to exactly one workspace. |
| **Standard Workspace** | A workspace that works exactly like Ascend does today — projects, tasks, notes. No AI memory features. | Workspace with `type = 'standard'`. No captures page, no entities, no chat, no oracle. Existing functionality only. |
| **Intelligence Workspace** | A workspace with the full memory layer — captures, entities, AI chat, and proactive monitoring. | Workspace with `type = 'intelligence'`. Enables all Memory Layer features. Content is embedded for semantic search. |
| **Capture** | An enhanced note with a type and date/time. Meeting notes, documents, thoughts — anything you want to remember. | Evolved from the existing `notes` table. Adds `capture_type` (meeting_note, document, media, thought), `occurred_at` (when it happened), and optional `project_id` (can exist at workspace level). |
| **Entity** | A thing you care about — a product, team, person, system, initiative. You name it and tag other things with it. | Record in the `entities` table with `name` and `description`, unique per workspace. Connected to captures, tasks, and projects via junction tables. |
| **Brain Dump** | A way to pour out everything you know in one go. The AI reads it and proposes entities and captures for you to review. | Multi-pass AI extraction pipeline using Claude Sonnet. Chunks large text, extracts entities (pass 1), then captures tagged with entities (pass 2). Progressive UI shows results as chunks complete. |
| **Embedding** | A mathematical fingerprint of a piece of content that captures its meaning, enabling search by concept rather than exact words. | 512-dimension vector generated by Voyage AI (`voyage-3-lite`). Stored in `embeddings` table using pgvector. Used for semantic similarity search via cosine distance. |
| **RAG Chat** | A chat interface where AI answers your questions using your actual data as context, with source citations. | Retrieval-Augmented Generation. User query → embed via Voyage → vector search → fetch source content → Claude (Haiku or Sonnet) generates answer with `[Source N]` citations. |
| **Oracle** | A monitoring system that proactively surfaces insights — things you might have missed. | Rule-based detectors (pure SQL) that run daily via Inngest cron. Checks for stale tasks, overdue items, velocity changes, entity gaps. Summarized by Haiku. Max 5 insights/day/workspace. |

## How It Works — User Journeys

### Journey 1: Setting Up Your Workspace (Brain Dump)

You just signed up or decided to try Intelligence Mode. Here's what happens:

1. **Create an Intelligence Workspace** — Click the workspace switcher in the sidebar, create a new workspace, and select "Intelligence" as the type. Give it a name like "Work" or "Product Team."

2. **Do a Brain Dump** — Open the Brain Dump page. A large text area appears with the prompt: "Write everything you know. Don't worry about structure — just dump it all."

3. **Write or paste everything** — Your company, the products, the people, the teams, what you're working on, what the problems are. Write as much or as little as you want. The system handles inputs from a single paragraph to 50,000+ words.

4. **Watch the AI work** — As you submit, the system processes your text in chunks. A progress bar shows "Processing chunk 3 of 7..." Results appear progressively — you don't wait for everything to finish.

5. **Review what the AI found** — The review screen shows two lists:
   - **Entities** (the things): "Acme Corp (company)", "ShipTrack (product)", "Sarah (person)", "Platform API (system)"
   - **Captures** (the knowledge): "Platform team structure", "API v3 migration plan", "Partner onboarding pain points"

   Each item has a checkbox, an edit button, and a description. If an entity already exists in your workspace, it shows "Already exists — link to existing" instead of creating a duplicate.

6. **Accept, edit, or reject** — Review each proposed item. Edit names and descriptions. Reject anything that's not useful. Then click "Create" — all accepted entities and captures are created in your workspace, properly tagged and connected.

7. **Repeat any time** — Brain dumps are not a one-time thing. Come back from a conference, start a new initiative, or just had a busy week? Do another brain dump. The system deduplicates against existing entities.

### Journey 2: Daily Work (Capturing & Tagging)

Now your workspace has context. Here's a typical day:

1. **Morning standup capture** — Before your 9am standup, click the quick capture button in the sidebar. Type "Standup — discussed API timeout, Sarah to follow up with partners." Hit enter. A capture is created under "Today" with the current time.

2. **Meeting notes with details** — After a longer meeting, go to `/captures` and create a new capture. Select type "Meeting Note", set the date/time to when the meeting happened, write detailed notes, and tag it with entities: "Platform API", "Sarah", "Partner Team."

3. **Tag a task** — While working on a task, open the task detail and add entity tags: "Platform API", "API v3 Migration." Now this task is linked to those entities.

4. **End of day review** — Go to `/captures`. See your daily journal:
   ```
   Today — Tuesday, Mar 11
     🗒 Standup (9:00 AM)
     📄 Competitor analysis doc (11:30 AM)
     🗒 Platform API deep dive (2:00 PM)
     💭 Idea: batch processing for partner imports
   ```
   Scroll down to see yesterday, the day before, and so on. Your entire week is visible, chronologically organized.

### Journey 3: Recalling Information (Chat)

Two weeks later, you need to remember something:

1. **Open chat** — Click the Chat button in the sidebar (or swipe up on mobile). A panel opens on the right side of the screen.

2. **Ask a question** — Type: "What do we know about the Platform API timeout issue?" The AI searches your captures, tasks, and entities, finds 3 relevant items, and responds:

   > "Based on your captures and tasks, the Platform API timeout issue was first discussed in your March 11 standup [Source 1]. Sarah was assigned to follow up with partners. A detailed investigation was captured in your 'Platform API deep dive' meeting note [Source 2], which identified a connection pooling bottleneck. There's an open task to fix the connection pool configuration [Source 3]."
   >
   > [Source 1: Standup — Mar 11] [Source 2: Platform API deep dive] [Source 3: Fix connection pool]

3. **Click a source** — Each source pill is clickable. Click "Platform API deep dive" and it navigates to that capture.

4. **Create a task from chat** — Type: "Create a task to write a postmortem for the API timeout." The AI proposes a structured task with title, description, and priority. Edit if needed, then confirm to create it.

5. **Switch models** — For a quick factual lookup, Haiku is fast and cheap. For deeper analysis ("Summarize all the Platform API issues this quarter and identify patterns"), switch to Sonnet — it's more powerful and produces richer analysis.

### Journey 4: Proactive Insights (Oracle)

You don't always know what to ask. The Oracle watches for patterns:

1. **Open chat panel** — At the top, above your conversation, you see insight cards:

   > ⚠️ **3 stale tasks detected**
   > "Investigate partner timeout", "Update rate limit headers", and "Review API docs" have been in-progress for over 7 days with no updates.
   > [View Tasks] [Dismiss]

   > ℹ️ **Platform API knowledge gap**
   > This entity has 5 active tasks but no new captures in 14 days. Consider adding recent meeting notes or status updates.
   > [View Entity] [Dismiss]

2. **Take action or dismiss** — Click "View Tasks" to go fix the stale tasks, or dismiss if it's not relevant right now.

3. **Configure** — Go to workspace settings to enable/disable the Oracle, toggle individual detectors (stale tasks, overdue items, velocity changes), or change the daily insight cap.

## Architecture Overview (Technical)

### Data Model

```
┌─────────────────────────────────────────────────────┐
│                    WORKSPACE                         │
│  id, name, type (standard|intelligence)              │
├─────────────────────────────────────────────────────┤
│                                                      │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐          │
│  │ PROJECTS │  │ CAPTURES │  │ ENTITIES │          │
│  │          │  │ (notes)  │  │          │          │
│  │ tasks    │  │ type     │  │ name     │          │
│  │ docs     │  │ date/time│  │ desc     │          │
│  │ members  │  │ content  │  │          │          │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘          │
│       │              │              │                │
│       └──────────────┼──────────────┘                │
│              entity junction tables                  │
│       (entity_projects, entity_tasks, entity_notes)  │
│                                                      │
│  ┌──────────────────────────────────────────┐       │
│  │            EMBEDDINGS (pgvector)          │       │
│  │  source_type, source_id, vector(512)     │       │
│  │  ← Voyage AI (voyage-3-lite)             │       │
│  └──────────────────────────────────────────┘       │
│                                                      │
│  ┌────────────────┐  ┌──────────────────┐           │
│  │ CHAT           │  │ ORACLE           │           │
│  │ conversations  │  │ insights         │           │
│  │ messages       │  │ config           │           │
│  │ ← Claude       │  │ ← Inngest cron   │           │
│  └────────────────┘  └──────────────────┘           │
│                                                      │
└─────────────────────────────────────────────────────┘
```

### Technology Stack

| Component | Technology | Why |
|-----------|-----------|-----|
| Vector store | Supabase pgvector | Already using Supabase. No new infrastructure. |
| Embeddings | Voyage AI (voyage-3-lite) | High quality, 512 dimensions, very cheap ($0.02/1M tokens) |
| Chat LLM (fast) | Claude Haiku 4.5 | Fast, cheap (~$0.001/turn). Good for factual lookups. |
| Chat LLM (deep) | Claude Sonnet 4.6 | Powerful reasoning. Good for analysis, summaries, pattern detection. |
| Brain dump extraction | Claude Sonnet 4.6 | Needs deep reasoning to extract structured data from unstructured text. |
| Oracle summaries | Claude Haiku 4.5 | Summarizing detector output is simple. Haiku is sufficient. |
| Background jobs | Inngest | Already using for task reminders. Cron for oracle, events for embedding. |
| Rate limiting | Upstash Redis | Already using. Add new bucket configs. |

### Embedding Pipeline

```
Content created/updated in intelligence workspace
        ↓
Hook fires Inngest event ("content/created" or "content/updated")
        ↓
Inngest function (background):
  1. Fetch content from DB (title + description/content)
  2. Compute SHA-256 hash of content
  3. Check if hash matches existing embedding → skip if unchanged
  4. Call Voyage AI → get 512-dim vector
  5. Upsert into embeddings table
        ↓
Content available for semantic search
```

### RAG Pipeline

```
User types question in chat
        ↓
1. Embed query via Voyage AI → 512-dim vector
2. Call match_embeddings() RPC → top 10 results above 0.6 threshold
3. Fetch full source content for each result (max 500 chars each)
4. Assemble system prompt:
   - "You are a workspace intelligence assistant for {workspaceName}"
   - Numbered source sections: [Source 1] type + title + content
   - "Answer ONLY from context. Cite sources. If insufficient, say so."
5. Include last 10 conversation messages for continuity
6. Call Claude (Haiku or Sonnet, user choice)
7. Parse response, save user + assistant messages to DB
8. Return response with source metadata for clickable pills
```

### Brain Dump Pipeline

```
User submits large text (potentially 50,000+ words)
        ↓
Client-side chunking:
  - Split on paragraph boundaries (~3000 words per chunk)
  - Include last paragraph of previous chunk as context overlap
        ↓
Pass 1 — Entity Extraction (sequential, per chunk):
  - Each chunk → Claude Sonnet → [{name, description, type}]
  - Merge across chunks: deduplicate by name (fuzzy match)
  - Progressive: show entities as each chunk completes
        ↓
Pass 2 — Capture Extraction (sequential, per chunk):
  - Each chunk + merged entity list → Claude Sonnet → [{title, content, type, entities[]}]
  - Progressive: show captures as each chunk completes
        ↓
Dedup against existing workspace entities:
  - Exact match → "Already exists ✓"
  - Fuzzy match → "Similar to X — merge?"
  - No match → create new
        ↓
User reviews all proposed entities + captures
  - Accept / reject / edit each item
        ↓
Bulk create: entities → captures → junction rows
```

Each MVP is independently deployable and adds real usable value. Each is tested before the next layer is added.

| MVP | Name | What It Delivers | Status |
|-----|------|-----------------|--------|
| **0** | Initiative Documentation | This document + plan alignment | ✅ Complete |

| **2** | Entities + Brain Dump | Entity tagging system, brain dump extraction, entity detail pages, **product label on all task views** | 🔲 Not Started |

### MVP Dependencies

```
MVP 1 (Workspaces + Captures)
  ↓
MVP 2 (Entities + Brain Dump)  ← requires workspaces + captures to exist
  ↓
MVP 3 (Embeddings + Search)    ← requires content to embed (captures, entities, tasks)
  ↓
MVP 4 (Chat + Task Creation)   ← requires embeddings for RAG retrieval
  ↓
MVP 5 (Oracle)                 ← requires task_events (MVP 3) + chat panel (MVP 4) to surface insights
```

## Mobile Responsiveness Strategy

All Memory Layer features must work on mobile. The approach:

| Feature | Desktop | Mobile |
|---------|---------|--------|
| Workspace switcher | Sidebar dropdown | Sidebar dropdown (same) |
| Captures / Daily journal | Full-width list | Full-width list (same layout) |
| Capture editor | Right Sheet panel | Bottom Sheet / full-screen |
| Quick capture | Sidebar button → popover | Bottom nav button → Drawer |
| Entity list | Grid/list view | List view |
| Product label on task rows | Pill/badge next to task title | Truncated pill (max 1, "+N" overflow) |
| Entity tag picker | Inline combobox | Inline combobox (same) |
| Brain dump input | Full-page textarea | Full-page textarea (same) |
| Brain dump review | Two-column layout | Single-column stacked |
| Chat panel | Right Sheet (fixed) | Bottom Drawer (swipe up) |
| Oracle insights | Cards in chat panel | Cards in chat drawer |

All uses Sheet (`src/components/ui/sheet.tsx`) and Drawer (`src/components/ui/drawer.tsx`) primitives — already responsive and production-tested in the existing app.
